# trace_evaluator

Pipeline that takes an LLM inference workload described by an MLSynth config, generates Chakra execution traces for it, and replays those traces through ASTRA-sim — either the **analytical** network backend or the packet-level **ns-3** backend — to produce system- and (for ns-3) packet-level logs for analysis.

```text
MLSynth config (configs/mlsynth/<model>.yaml)
        │
        ▼  generate_traces.sh  (ns-3 flow only, phase 1/2)
Chakra traces (output/mlsynth/<model>/)
        │
        ▼  generate_log_ns3.sh (phase 2/2)         or   generate_log_analytical.sh (single phase, runs MLSynth itself)
ASTRA-sim system-layer logs (output/astra_logs/...)  +  ns-3 packet-level logs (output/ns3/...)
```

For ns-3-specific fabric config generation (`config_generator.py`, the `__MODEL__`/tag scheme, physical/logical topology files), see [configs/astra_sim/ns3/documentation/README.md](configs/astra_sim/ns3/documentation/README.md). This file covers the overall pipeline and the `output/` layout shared by both backends.

---

## Scripts

| Script | Backend | What it does |
| --- | --- | --- |
| `generate_traces.sh <model>` | both (ns-3 flow) | Phase 1/2. Runs MLSynth once for `<model>`, writing `output/mlsynth/<model>/`. Run it once before one or more `generate_log_ns3.sh` calls for that model — running MLSynth concurrently for the *same* model would corrupt the shared trace directory. Safe to run once and reuse across many ns-3 fabric configs, including in parallel. |
| `generate_log_ns3.sh <model> <ns3_subdir> [output_dir_name]` | ns-3 | Phase 2/2. Requires traces from `generate_traces.sh` to already exist for `<model>`. Resolves the `__MODEL__` placeholder in the fabric config at `configs/astra_sim/ns3/<ns3_subdir>/config.txt` into a temporary copy, then launches the ns-3 ASTRA-sim binary. `<ns3_subdir>` is the fabric config's path relative to `configs/astra_sim/ns3/` (e.g. `bandwidth_sweep/T2_bx100_dcqcn_buf32`). |
| `generate_log_analytical.sh <model> [output_dir_name]` | analytical | Single phase: runs MLSynth itself (no traces required beforehand) and then the analytical ASTRA-sim binary. No fabric config / topology files involved — network parameters come from `configs/astra_sim/analytical/network2D.yml`. |
| `run_all.sh <model> <ns3_subdir> [output_dir_name]` | ns-3 | Convenience wrapper: calls `generate_traces.sh` then `generate_log_ns3.sh` sequentially, for a single one-off job. For sweeps (several `ns3_subdir`s for the same model, possibly in parallel) call the two scripts separately instead, so MLSynth only runs once. |

All scripts accept their arguments positionally, or prompt interactively when run with a terminal attached and an argument omitted; without a terminal (e.g. in CI or a background job) all required arguments must be passed explicitly.

## `output/` directory structure

Everything under `output/` is generated by the scripts above — nothing there is meant to be hand-edited, and folders can be deleted freely to force a clean re-run.

The MLSynth traces are keyed by **model only** and shared across fabric configs. The ASTRA-sim/ns-3 logs are keyed by **model, then fabric config** (`<ns3_subdir>`, i.e. `[<subdir>/]<tag>` — the same relative path as the fabric config under `configs/astra_sim/ns3/`), so the same fabric config can be replayed against several models, and the same model against several fabric configs, without collisions:

```text
output/
    mlsynth/
        <model>/
            comm_groups.json
            et/
                <model>.<rank>.et             # one Chakra trace per NPU/rank

    astra_logs/
        analytical/
            <model>/                          # generate_log_analytical.sh default
                log.log
                err.log
                stats_sys<rank>.csv
        <model>/
            [<subdir>/]<tag>/                 # generate_log_ns3.sh default (ns-3 flow); one folder per fabric config
                log.log
                err.log
                stats_sys<rank>.csv

    ns3/
        <model>/
            [<subdir>/]<tag>/                 # packet-level (ns-3) logs, mirrors astra_logs/<model>/[<subdir>/]<tag>/
                fct.txt
                pfc.txt
                qlen.txt
                traces.tr
```

Notes:

* `output/mlsynth/<model>/` is produced once per model (`generate_traces.sh`, or internally by `generate_log_analytical.sh`) and reused by every later ns-3 run for that model.
* `output/astra_logs/<model>/[<subdir>/]<tag>/` and `output/ns3/<model>/[<subdir>/]<tag>/` are both written by `generate_log_ns3.sh`. The `astra_logs` folder is `rm -rf`'d and recreated on every run to avoid mixing logs from different attempts; the `ns3` folder is only `mkdir -p`'d, so leftover files from an older run at the same path are not cleared automatically.
* The third, optional argument to `generate_log_ns3.sh`/`generate_log_analytical.sh`/`run_all.sh` overrides only the `astra_logs` destination name (`OUTPUT_DIR_NAME`, default `<model>/<ns3_subdir>` or `analytical/<model>`); it does **not** affect the `ns3/` destination, which is always `<model>/<ns3_subdir>` — this keeps packet-level outputs deterministically tied to (model, fabric config) regardless of how you label the system-layer log folder.
* Two runs sharing **both** the same model **and** the same fabric config write to the same `ns3/<model>/<ns3_subdir>/` folder and must not run concurrently.
* `generate_log_analytical.sh` never touches `output/ns3/` — the analytical backend has no packet-level output.

## `configs/` directory structure

```text
configs/
    mlsynth/                       # one YAML per model, input to MLSynth (workload description)
    astra_sim/
        system/                    # ASTRA-sim system configs (e.g. h100_1D.json, h100_2D.json)
        no_remote_memory.json
        logging_config.toml
        analytical/                # analytical backend: network2D.yml (no topology files, no fabric configs)
        ns3/                       # ns-3 backend: fabric configs, see its own README
            documentation/README.md
```
