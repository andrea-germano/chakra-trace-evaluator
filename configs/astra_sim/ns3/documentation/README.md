# ns-3 Backend — configuration (trace_evaluator)

These files configure the **ns-3** (packet-level) backend of ASTRA-sim, used instead of the analytical one to simulate the network on Chakra traces generated by MLSynth.

The new workflow has become dynamic: it involves using a Python script to set the desired parameters and generate a specific working folder (identified by a *tag*), inside which the user must then manually add the files related to the physical and logical topologies. All paths generated in the config are absolute, meaning the simulation binary can be launched from any directory.

---

## Directory Structure and Workflow

```text
ns3/
    gen_config.py              # Interactive script to generate the config file[cite: 2]
    config.template.txt    # Simulator configuration template[cite: 2]
    additional_configs/               # Folder containing static input requests[cite: 2]
        add_flow.txt               # Static input: manual flows (default empty/0)[cite: 2]
        trace_link.txt              # Static input: nodes to monitor (default 0)[cite: 2]
    config_documentation.txt   # Documentation of individual ns-3 parameters[cite: 2]
    <tag>/                     # Folder dynamically generated by the script (e.g., "test")[cite: 2]
        config.txt             # Final file for ns-3 generated by the Python script[cite: 2]
        physical_topology.txt  # TO BE ADDED MANUALLY (hardware topology)[cite: 2]
        logical_topology.json  # TO BE ADDED MANUALLY (logical collectives)[cite: 2]

```

The simulation outputs (e.g., `fct.txt`, `pfc.txt`, `qlen.txt`, `mix.tr`) will be automatically redirected to the directory `/home/andre/tesi/trace_evaluator/output/ns3/<tag>/`.

## How to start the configuration

1. **Run the Python script:**
* From the terminal, run `python gen_config.py` and answer the interactive prompts.


* You will be asked for a `tag` (the name the test folder will take).


* You can choose the congestion control protocol (e.g., dcqcn, hpcc, dctcp, hpcc-pint) and other parameters like buffer size and enabling the packet trace.




2. **Config generation:**
* The script will replace the environment variables in the base template, creating the `ns3/<tag>/config.txt` file with the correct settings.




3. **Manual addition of topologies:**
* As suggested by the script's end-of-execution message, you will need to navigate into the newly created `<tag>/` folder and place the necessary files for the simulation inside it: `physical_topology.txt` and `logical_topology.json`.

## Explanation of main files

### `physical_topology.txt` (Physical topology)

This file (which replaces the previous `simple_topology.txt`) defines the hardware layout and must be placed inside the tag-specific folder.

* **Strict format**: It does not accept comments or extra lines.


* **Line 1**: `<total_nodes> <num_switches> <num_links>`.


* **Line 2**: List of IDs assigned to the switches.


* **Subsequent lines**: Link configuration in the format `<src> <dst> <bandwidth> <delay> <error_rate>`.


* **Node Numbering**: Compute nodes (NPUs) must strictly have low IDs starting from `0` up to `N-1`, followed by the switches.


* **Central Constraint**: The total number of NPUs used (`total_nodes` - `num_switches`) must strictly match the number of `.et` files generated by MLSynth.


* **Bandwidth Constraint**: Any link capacity specified here (e.g., `200Gbps`) must exist within the ECN maps configured in `config.template.txt`, otherwise the simulator will crash unexpectedly.



### `logical_topology.json` (Logical topology)

Logically separates how collective operations map onto the physical hardware network. It must be placed in the tag-specific folder.

* **JSON Format**: Contains the dimensional array, for example `{ "logical-dims": ["6"] }` for a flat 1-D dimension with 6 nodes.


* **Product of dimensions**: Multiplying the numbers in the array must yield the exact total number of NPUs in the simulation.


* **Length Constraint**: The number of dimensions in this array is constrained to equal the length defined in the `*-implementation` lists of the system config.



### `config.txt` (The simulator file)

It is compiled by `gen_config.py` and is located in the tag folder, ready to be used by ns-3.

* The CC string (e.g., `hpcc-pint`) is translated by the script into the corresponding documented numeric ID, automatically handling the mapping without having to memorize it (e.g., dcqcn = 1, hpcc-pint = 10, etc.).


* Redirects the search for the topology file to the path `/__TAG__/physical_topology.txt` using fixed paths.



### `additional_configs/add_flow.txt` & `additional_configs/trace_link.txt`

These are **Input** files kept unchanged within the `additional_configs/` folder.

* `add_flow.txt = 0` → Zero manual flows inserted (ASTRA-sim will autonomously inject its own workloads).


* `trace_link.txt = 0` → No specific node being monitored.

## Representing different network topologies

The physical topology resides entirely in the new `physical_topology.txt` within the generated folder. By changing the three header lines and the link list, it is possible to model and test different network scenarios:

### Single-switch

Ideal for a sanity check, places all NPUs on a single large switch.

```text
9 1 8            # Example: 8 NPUs + 1 switch[cite: 2]
8                # Switch ID (node 8)[cite: 2]
8 0 200Gbps 0.005ms 0[cite: 2]
... (continue up to link "8 7")[cite: 2]

```

When updating this network, the variable in the logical JSON file must consequently be updated, changing it to `["8"]`.

### Fat-tree / Clos (two tiers)

Requires two tiers of switches (Leaf/ToR for the NPUs and upper Spine for interconnection). It is the suitable structure to study cross-node congestion bottlenecks.
It usually requires a two-dimensional logical topology mapping (e.g., `["4","2"]` intra-ToR and inter-ToR) with consequent updates to the JSON file and ASTRA-sim system config.

### Ring (switchless)

All NPUs are daisy-chained directly to each other, leaving the second line of the topology file empty (`num_switch = 0`):

```text
8 0 8[cite: 2]
0 1 25Gbps 0.005ms 0[cite: 2]
1 2 25Gbps 0.005ms 0[cite: 2]
...[cite: 2]
7 0 25Gbps 0.005ms 0[cite: 2]

```

## Summary Checklist

To be executed in case of new MLSynth configurations or modifications to the topological layout:

1. Pre-count the input NPUs: `tp_p * pp_p + tp_d * pp_d` (printed at the end of the MLSynth script).


2. Run the Python script `gen_config.py` and define the new workspace by filling in the parameters.


3. Create or copy the `physical_topology.txt` file into the newly generated folder. Verify that line 1, the switch IDs, and the links are updated.


4. Create `logical_topology.json` in the same folder and ensure that the product of the dimensions inside it gives exactly the previously calculated total of NPUs.


5. Scale the same dimensions onto the `*-implementation` lists of the system config to align them.


6. Validate that all bandwidths of your links (in the physical topology file) have a mapping in the `KMAX/KMIN/PMAX` lines in `config.template.txt` prior to generation (otherwise, update the template).
