# ns-3 Backend — configuration (trace_evaluator)

These files configure the **ns-3** (packet-level) backend of ASTRA-sim, used instead of the analytical one to simulate the network on Chakra traces generated by MLSynth.

The workflow is dynamic and split into two stages: `config_generator.py` builds a reusable, **model-agnostic** fabric config (topology, congestion control, buffer size, ...) identified by a derived *tag*; `generate_log_ns3.sh` (in the repo root) picks a model and a fabric config at run time, resolves the model into the config, and launches the ns-3 binary. All paths generated in the config are absolute, so the simulation binary can be launched from any directory.

---

## Directory Structure and Workflow

```text
ns3/
    config_generator.py        # Interactive script that generates a fabric config
    config.template.txt        # Simulator configuration template
    additional_configs/        # Static input files, shared by every fabric config
        add_flow.txt               # Static input: manual flows (default empty/0)
        trace_link.txt              # Static input: nodes to monitor (default 0)
    documentation/
        README.md                  # This file
        file_config_documentation.txt  # Documentation of individual ns-3 parameters
    [<subdir>/]<tag>/           # Folder dynamically generated by the script
        config.txt                  # Final file for ns-3, run-agnostic (__RUN_DIR__ placeholder)
        physical_topology.txt       # TO BE ADDED MANUALLY (hardware topology)
        logical_topology.json       # Generated automatically (flat 1-D over all NPUs)
        manifest.json                # Generated automatically: full record of this run's parameters
```

`<subdir>` is optional and only exists if you answer the "subdirectory" prompt (e.g. `buffer_sweep_T1/`, `bandwidth_sweep/`); it lets you group related fabric configs (a sweep) instead of dumping everything flat under `ns3/`. `<tag>` is never typed directly — it is derived from the parameters you enter (see below).

A fabric config produced this way is **reused across models and experiments**: the same `<subdir>/<tag>/config.txt` can be replayed for `bert_base`, `gpt2`, a bandwidth sweep, an incast sweep, etc. without regeneration, because no output-dir name is baked in at generation time — it is substituted at run time (see next section).

---

## How to generate a fabric config

1. **Run the Python script:**
   * From the `ns3/` directory, run `python config_generator.py` and answer the interactive prompts (press Enter to keep the shown default).
   * Prompts, in order: subdirectory (optional), topology label (free text, e.g. `2a`, `T2`), number of NPUs, fabric bandwidth `Bx` in Gbps, congestion control protocol (`dcqcn`, `hpcc`, `timely`, `dctcp`, `hpcc-pint`, `none`), buffer size, and finally instrumentation options (packet trace, qlen monitor start/end — these are recorded in `manifest.json` but do not affect the tag).

2. **Tag derivation and config generation:**
   * The tag is built automatically as `<topology_label>_bx<Bx>_<cc>_buf<buffer_size>` (e.g. `T2_bx100_dcqcn_buf32`).
   * The script replaces the placeholders in `config.template.txt` and writes `config.txt`, `logical_topology.json`, and `manifest.json` into the new `[<subdir>/]<tag>/` folder. The CC name is translated into its numeric ID automatically (e.g. dcqcn = 1, hpcc-pint = 10 — see `CC_MODE` in the script).
   * `config.txt` deliberately keeps a `__RUN_DIR__` placeholder in every output-file line — it is **not** resolved here, because a fabric config is run-agnostic (see "Run-keyed outputs" below).

3. **Manual addition of the physical topology:**
   * As suggested by the script's end-of-execution message, navigate into the newly created `<tag>/` folder and drop `physical_topology.txt` inside it. `logical_topology.json` and `manifest.json` are already generated for you.

## Run-keyed outputs and `__RUN_DIR__` resolution

`config.txt`'s `TRACE_OUTPUT_FILE`, `FCT_OUTPUT_FILE`, `PFC_OUTPUT_FILE`, and `QLEN_MON_FILE` all point under:

```text
/home/andre/tesi/trace_evaluator/output/ns3/__RUN_DIR__/
```

`__RUN_DIR__` is only resolved by `generate_log_ns3.sh` (in the repo root, **not** in this folder), just before launching the ns-3 binary, into a per-run temporary copy of the config:

```bash
./generate_log_ns3.sh <model_file_name> <ns3_subdir> [output_dir_name]
# example:
./generate_log_ns3.sh bert_base bandwidth_sweep/T2_bx100_dcqcn_buf32
```

where `<ns3_subdir>` is `[<subdir>/]<tag>` relative to `configs/astra_sim/ns3/` (i.e. the path to the folder containing `config.txt`, without the `ns3/` prefix), and `[output_dir_name]` defaults to `<model_file_name>/<ns3_subdir>` when omitted.

`__RUN_DIR__` is substituted with `output_dir_name` **verbatim**, so ns-3 packet-level outputs always land at `output/ns3/<output_dir_name>/`, mirroring `output/astra_logs/<output_dir_name>/` exactly — regardless of `<ns3_subdir>`'s own folder path under `configs/astra_sim/ns3/`. As a result:

* the **same** fabric config replayed with a **different** `output_dir_name` no longer collides (each gets its own `output/ns3/<output_dir_name>/...` subtree);
* two runs sharing the **same** `output_dir_name` still write to the same ns-3 output folder and must not run concurrently.

`generate_log_ns3.sh` refuses to run (fails fast) if `config.txt` has no `__RUN_DIR__` placeholder, so a fabric config generated before this scheme existed must be regenerated with the current `config_generator.py`/`config.template.txt`, or patched by hand to replace the model/tag segment with `__RUN_DIR__` in every `*_OUTPUT_FILE`/`QLEN_MON_FILE` line.

**Consistency rule for `TOPOLOGY_FILE`/`FLOW_FILE`/`TRACE_FILE`:** the `[<subdir>/]<tag>` segment they use (pointing at the config's own folder) must match the config's **actual, current** folder path — this is unrelated to `__RUN_DIR__` and still governs where `physical_topology.txt` etc. are read from. If you rename a `<subdir>/<tag>/` folder after generation, update `config.txt` accordingly (or just regenerate it).

## Explanation of main files

### `physical_topology.txt` (Physical topology)

Defines the hardware layout and must be placed inside the tag-specific folder.

* **Strict format**: It does not accept comments or extra lines.
* **Line 1**: `<total_nodes> <num_switches> <num_links>`.
* **Line 2**: List of IDs assigned to the switches.
* **Subsequent lines**: Link configuration in the format `<src> <dst> <bandwidth> <delay> <error_rate>`.
* **Node Numbering**: Compute nodes (NPUs) must strictly have low IDs starting from `0` up to `N-1`, followed by the switches.
* **Central Constraint**: The total number of NPUs used (`total_nodes` - `num_switches`) must strictly match the number of `.et` files generated by MLSynth for the model being simulated.
* **Bandwidth Constraint**: Any link capacity specified here (e.g., `200Gbps`) must exist within the ECN maps (`KMAX_MAP`/`KMIN_MAP`/`PMAX_MAP`) configured in `config.template.txt`, otherwise the simulator will crash unexpectedly. `config_generator.py` only *warns* about this (it does not block you) if the `Bx` you typed is outside the known set.

### `logical_topology.json` (Logical topology)

Generated automatically by `config_generator.py` as a flat 1-D dimension over all NPUs, e.g. `{ "logical-dims": ["8"] }`. ASTRA-sim rebuilds the real collective structure from `comm_groups.json` at run time, so this file only needs to satisfy two formal constraints:

* **Product of dimensions**: Multiplying the numbers in the array must yield the exact total number of NPUs in the simulation.
* **Length Constraint**: The number of dimensions in this array must equal the length of the `*-implementation` lists in the ASTRA-sim system config (e.g. `configs/astra_sim/system/h100_1D.json`).

If you need a non-flat logical topology, edit this file by hand after generation.

### `manifest.json` (Run manifest)

Generated automatically by `config_generator.py`. The authoritative, complete record of the fabric config's parameters (topology label, node count, bandwidth, CC, buffer size, instrumentation flags, generation timestamp) — the folder name/tag is just the human-readable handle, this file is the source of truth if you need to recall exactly how a config was built.

### `config.txt` (The simulator file)

Compiled by `config_generator.py`, run-agnostic (`__RUN_DIR__` placeholder in every output path), and located in the tag folder. Resolved into a temporary, run-specific copy by `generate_log_ns3.sh` right before invoking ns-3 (see "Run-keyed outputs" above).

* The CC string (e.g., `hpcc-pint`) is translated by the script into the corresponding documented numeric ID automatically (e.g., dcqcn = 1, hpcc-pint = 10 — see `file_config_documentation.txt` for the full list).
* `TOPOLOGY_FILE`/`FLOW_FILE`/`TRACE_FILE` use fixed, absolute paths into this config's own folder and into `additional_configs/`.

### `additional_configs/add_flow.txt` & `additional_configs/trace_link.txt`

These are **Input** files kept unchanged within the `additional_configs/` folder, shared by every fabric config.

* `add_flow.txt = 0` → Zero manual flows inserted (ASTRA-sim will autonomously inject its own workloads).
* `trace_link.txt = 0` → No specific node being monitored.

## Representing different network topologies

The physical topology resides entirely in the `physical_topology.txt` you drop into the generated folder. By changing the three header lines and the link list, it is possible to model and test different network scenarios:

### Single-switch

Ideal for a sanity check, places all NPUs on a single large switch.

```text
9 1 8            # Example: 8 NPUs + 1 switch
8                # Switch ID (node 8)
8 0 200Gbps 0.005ms 0
... (continue up to link "8 7")
```

When using this network, `logical_topology.json` must have `["8"]` (already the default `config_generator.py` output for `num_nodes=8`).

### Fat-tree / Clos (two tiers)

Requires two tiers of switches (Leaf/ToR for the NPUs and upper Spine for interconnection). It is the suitable structure to study cross-node congestion bottlenecks.
It usually requires a two-dimensional logical topology mapping (e.g., `["4","2"]` intra-ToR and inter-ToR), which means hand-editing `logical_topology.json` after generation, with consequent updates to the ASTRA-sim system config.

### Ring (switchless)

All NPUs are daisy-chained directly to each other, leaving the second line of the topology file empty (`num_switch = 0`):

```text
8 0 8
0 1 25Gbps 0.005ms 0
1 2 25Gbps 0.005ms 0
...
7 0 25Gbps 0.005ms 0
```

## Summary Checklist

To be executed in case of new MLSynth configurations or modifications to the topological layout:

1. Pre-count the input NPUs: `tp_p * pp_p + tp_d * pp_d` (printed at the end of the MLSynth script), or the number of `.et` files under `output/mlsynth/<model>/et/`.
2. Run `python config_generator.py` and define the new fabric config by filling in the parameters (subdirectory, topology label, node count, bandwidth, CC, buffer size).
3. Create or copy the `physical_topology.txt` file into the newly generated `[<subdir>/]<tag>/` folder. Verify that line 1, the switch IDs, and the links are updated, and that the NPU count matches step 1.
4. `logical_topology.json` is already generated for the flat 1-D case; hand-edit it only if you need a multi-dimensional mapping, keeping the product of its dimensions equal to the NPU count from step 1.
5. Scale the same dimensions onto the `*-implementation` lists of the ASTRA-sim system config to align them.
6. Validate that all bandwidths of your links (in the physical topology file) have a mapping in the `KMAX/KMIN/PMAX` lines in `config.template.txt` prior to generation (otherwise, update the template).
7. Generate the MLSynth traces for the model once (`./generate_traces.sh <model_file_name>`), then launch the ns-3 run with `./generate_log_ns3.sh <model_file_name> [<subdir>/]<tag>`.
